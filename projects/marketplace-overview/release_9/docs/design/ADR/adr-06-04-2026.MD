# Switch Marketplace Overview Screen Enablement to `ui_config` Query

## Status

Accepted

## Context

Release_9 navigation link visibility for Marketplace Overview was previously tied to static subscription-type mapping. The requirement now is to fetch enabled Brand Protector screens directly from `i2oretail.ui_config` for each tenant (`org_id = 277` example) using:

```sql
SELECT x.*
FROM i2oretail.ui_config x
WHERE org_id = 277
  AND property_cd IN (
    'BrandProtector',
    'BrandProtector.Enforcement_Center',
    'BrandProtector.BrandViolation.BrandViolations'
  )
ORDER BY x.property_cd;
```

This logic must be wrapped in one API in `i2o-reseller` so frontend consumers do not reimplement DB-driven enablement rules.

## Decision Drivers

- Tenant-specific configuration must come from DB config (`ui_config`) instead of static code mapping.
- Navigation and screen enablement must be centrally resolved in backend (`i2o-reseller`).
- Contract must explicitly support Brand Protector, Enforcement Center, and Brand Violations keys.
- Avoid frontend drift caused by duplicated enablement logic.

## Considered Options

- Option 1: Keep `organization.subscription_type` ID mapping for link visibility.
- Option 2: Read `ui_config` in frontend and compute link visibility in UI layer.
- Option 3: Read `ui_config` in `i2o-reseller` and expose resolved enablement via `GET /marketplace-overview/config`.

## Decision Outcome

Chosen option: "Option 3", because it keeps tenant enablement logic authoritative in backend, uses the requested `ui_config` query contract, and preserves a single API contract for frontend consumption.

### Positive Consequences

- Per-org enablement now reflects DB configuration without code redeploy for mapping updates.
- Frontend receives resolved booleans (`enabledModules`) and does not need DB/key knowledge.
- Screen enablement behavior is auditable and testable via one API endpoint.

### Negative Consequences

- Requires reliable seeding/maintenance of `ui_config` property keys per environment.
- Incorrect or missing property codes can hide links unexpectedly unless validation checks exist.

## Pros and Cons of the Options

### Option 1

- Good: No API contract changes.
- Bad: Static mapping diverges from tenant configuration and requested query contract.

### Option 2

- Good: Reduces backend transformation logic.
- Bad: Leaks config semantics to frontend and duplicates enablement rules across clients.

### Option 3

- Good: Single backend contract, centralized logic, tenant-aware behavior.
- Bad: Adds dependency on `ui_config` data quality and backend mapping tests.
