# Switch Active Subscription Source to `ui_config` for Marketplace Overview

## Status

Accepted

## Context

Marketplace Overview active subscription cards were previously modeled from `org_market_mapping`. The updated requirement is to source active subscription card enablement/configuration from `i2oretail.ui_config` as well, so config and navigation decisions come from one org-scoped configuration source family.

Target query pattern:

```sql
SELECT x.*
FROM i2oretail.ui_config x
WHERE x.org_id = :org_id
  AND x.property_cd LIKE 'BrandProtector.MarketplaceOverview.ActiveSubscription.%'
ORDER BY x.property_cd;
```

`GET /marketplace-overview/config` in `i2o-reseller` remains the single wrapper API that resolves:
- active subscriptions from `ui_config`
- screen enablement from `ui_config` property codes
- brand and enforcement enrichment from existing reference tables

## Decision Drivers

- Keep active-card and navigation enablement in consistent org-level configuration storage.
- Reduce split-source logic between `org_market_mapping` and `ui_config`.
- Preserve one backend API contract for frontend consumers.
- Support tenant-level flexibility via config rows without additional client-side mapping logic.

## Considered Options

- Option 1: Keep active subscriptions in `org_market_mapping`; use `ui_config` only for navigation.
- Option 2: Use `ui_config` for active subscriptions and navigation, resolved in backend config API.
- Option 3: Hybrid fallback with priority to `org_market_mapping` and optional `ui_config` overrides.

## Decision Outcome

Chosen option: "Option 2", because it aligns with the requirement to source active subscription data from `ui_config`, keeps config logic centralized, and avoids mixed-source ambiguity at runtime.

### Positive Consequences

- Unified configuration source family for active subscriptions and screen enablement.
- Backend contract becomes clearer for frontend (`/marketplace-overview/config` remains canonical).
- Per-org adjustments can be applied through configuration rows.

### Negative Consequences

- Active card composition now depends on `ui_config` row quality and JSON payload consistency.
- Requires validation/guardrails for property patterns and `filter_metadata` schema.

## Pros and Cons of the Options

### Option 1

- Good: No refactor for active-card data source.
- Bad: Keeps split ownership and inconsistent configuration behavior.

### Option 2

- Good: Single source family and deterministic backend composition path.
- Bad: More dependency on config data governance.

### Option 3

- Good: Safer migration path during transition.
- Bad: Adds precedence complexity and testing overhead.
