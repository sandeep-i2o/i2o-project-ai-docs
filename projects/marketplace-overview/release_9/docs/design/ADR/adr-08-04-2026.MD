# Scope Marketplace Config Brands by `brand_master.marketplace_ids`

## Status

Accepted

## Context

`GET /marketplace-overview/config` was returning the same org-level `brand_master` list for every active subscription card.  
Release_9 cards are marketplace+region specific, and `brand_master.marketplace_ids` already models marketplace-level brand applicability.  
Without marketplace scoping, the API response is functionally incorrect for multi-marketplace orgs.

## Decision Drivers

- Ensure each subscription card shows only brands valid for its marketplace+region.
- Reuse existing schema contract (`brand_master.marketplace_ids`) with no schema migration.
- Keep backward compatibility for the root `brands[]` list expected by existing frontend filter wiring.

## Considered Options

- Option 1: Keep org-level brand query and apply filtering only in frontend.
- Option 2: Scope backend brand lookup per active `marketplace_id` using `marketplace_id = ANY(brand_master.marketplace_ids)`.
- Option 3: Introduce a new normalized join table between brands and marketplaces.

## Decision Outcome

Chosen option: "Option 2", because it fixes correctness in the API contract immediately, uses existing data modeling, and avoids schema churn in release_9.

### Positive Consequences

- `subscriptions[].brands` and `subscriptions[].brandCount` are now marketplace-region accurate.
- API behavior aligns with existing `brand_master.marketplace_ids` ownership semantics.
- Root-level `brands[]` can remain a de-duplicated union across active subscriptions for frontend compatibility.

### Negative Consequences

- Backend performs per-marketplace brand lookups during config assembly.
- Legacy data rows with null/empty `marketplace_ids` are excluded from marketplace-scoped brand lists.

## Pros and Cons of the Options

### Option 1

- Good: No backend code change.
- Bad: Leaves incorrect API behavior and pushes domain correctness to UI.

### Option 2

- Good: Correct backend contract and clear domain ownership in one place.
- Bad: Additional query calls per active subscription.

### Option 3

- Good: Fully normalized relational model.
- Bad: Requires migration, backfill, and broader rollout risk not needed for this fix.

## More Information

- Impacted architecture sections: 4.1, 6.1, 7.1, 8.1, 15, 18.
- Impacted module/API: `i2o-reseller` `GET /marketplace-overview/config`.
- Impacted data contract: `brand_master.marketplace_ids` interpreted as authoritative marketplace membership.
