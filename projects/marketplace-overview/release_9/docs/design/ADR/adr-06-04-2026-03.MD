# Revert Active Subscription Sourcing to Legacy Mapping API

## Status

Accepted

## Context

Release_9 briefly moved active subscription card sourcing to `ui_config` (ADR-006). The current directive is to use the older mapping path again:
- marketplace + region mapping from `org_market_mapping` (legacy API path)
- brands from `brand_master`
- enforcement accounts from `account` mapped via `account_brand` with `org_id` scope
- only screen enablement remains in `ui_config` (`BrandProtector`, `BrandProtector.Enforcement_Center`, `BrandProtector.BrandViolation.BrandViolations`)

This update keeps `GET /marketplace-overview/config` as the single API wrapper, but changes the active-subscription source back to legacy mapping.

## Decision Drivers

- Preserve compatibility with existing/older API behavior for active marketplace-region mapping.
- Reduce release risk by avoiding active-subscription schema migration in `ui_config`.
- Keep screen gating centralized in `ui_config` as requested.
- Maintain deterministic backend composition for frontend.

## Considered Options

- Option 1: Keep active subscriptions in `ui_config` (ADR-006 path).
- Option 2: Revert active subscriptions to `org_market_mapping`; keep only screen enablement in `ui_config`.
- Option 3: Dual-source fallback (`ui_config` first, `org_market_mapping` fallback).

## Decision Outcome

Chosen option: "Option 2", because it directly matches the requested implementation approach and lowers migration risk for release_9.

### Positive Consequences

- Reuses known marketplace/region mapping flow with minimal backend change risk.
- Clear separation: subscription mapping data from legacy tables, screen gates from `ui_config`.
- Keeps frontend contract stable through one config endpoint.

### Negative Consequences

- Restores split-source configuration (mapping vs screen enablement).
- Requires extra validation to keep mapping data and screen gates logically aligned.

## Pros and Cons of the Options

### Option 1

- Good: Single-source-family concept for config.
- Bad: Higher migration risk for release_9 and dependency on new `ui_config` payload modeling.

### Option 2

- Good: Lowest risk path; aligns with current directive and existing legacy behavior.
- Bad: Keeps split-source model instead of full config unification.

### Option 3

- Good: Potentially resilient during transition.
- Bad: Adds complexity, precedence ambiguity, and larger test matrix.

## More Information

- Supersedes: ADR-006
- Affected API: `GET /marketplace-overview/config`
- Active subscription query:
  - `org_market_mapping` + `marketplace`
- Brand query:
  - `brand_master`
- Enforcement account query:
  - `account` + `account_brand` (org-scoped)
- Screen enablement query:
  - `ui_config` with property codes:
    - `BrandProtector`
    - `BrandProtector.Enforcement_Center`
    - `BrandProtector.BrandViolation.BrandViolations`
