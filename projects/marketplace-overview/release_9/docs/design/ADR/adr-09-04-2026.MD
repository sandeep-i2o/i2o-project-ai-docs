# Re-enable Unsubscribed Metrics via `marketplace_unsubscribed_metrics`

## Status

Accepted

## Context

Release_9 architecture had previously removed backend sourcing for unsubscribed marketplace metrics and used UI placeholders.  
Current product direction requires unsubscribed card metrics (`total products`, `total listings`, `total resellers`) to be fetched from a new PostgreSQL table, while also carrying audit/trial state for CTA behavior.

## Decision Drivers

- Deliver backend-backed unsubscribed marketplace metrics in release_9.
- Define an explicit and stable table contract for org+marketplace metrics and audit/trial state.
- Minimize frontend round-trips by returning unsubscribed metrics in the existing `GET /marketplace-overview/config` response.

## Considered Options

- Option 1: Keep placeholder-only UI values (`##`) and defer backend sourcing.
- Option 2: Add new table `marketplace_unsubscribed_metrics` and include data in `GET /marketplace-overview/config`.
- Option 3: Add a separate endpoint for unsubscribed metrics and keep `/config` unchanged.

## Decision Outcome

Chosen option: "Option 2", because it introduces the required data contract with minimal API surface expansion and keeps page-load orchestration simple.

### Positive Consequences

- Unsubscribed cards now render deterministic backend metrics for `totalProducts`, `totalListings`, and `totalResellers`.
- Audit/trial state fields (`audit_requested`, `trial_initiated`, `audit_status`, `audit_type`, `audit_requested_time`, `audit_report_gcs_link`) are available in one canonical table.
- Existing `/config` aggregation remains the single page bootstrap contract.

### Negative Consequences

- Requires database migration and backfill before production readiness.
- Adds schema governance overhead for temporary columns (`temp1`, `temp2`) until they are replaced or removed.

## Pros and Cons of the Options

### Option 1

- Good: No backend migration or API changes.
- Bad: Continues placeholder-only UX and fails the updated metrics requirement.

### Option 2

- Good: Meets requirement with explicit schema and single-config response contract.
- Bad: Requires data pipeline readiness and operational monitoring for freshness.

### Option 3

- Good: Keeps `/config` payload smaller and separates concerns.
- Bad: Adds one more page-load API call and orchestration complexity for frontend state.

## More Information

- New table contract fields: `org_id`, `marketplace_id` (FK), `total_products`, `total_listings`, `total_resellers`, `audit_requested`, `trial_initiated`, `temp1`, `temp2`, `audit_status`, `audit_type`, `audit_requested_time`, `audit_report_gcs_link`.
- Impacted architecture sections: 1.4, 2, 4.1, 6.1, 7.2, 7.4, 7.5, 8.1, 8.2, 9.4, 10.4, 10.5, 15, 18.
- Impacted API: `GET /marketplace-overview/config`.
- Impacted backend modules: `MarketplaceOverviewConfigService`, `MarketplaceOverviewConfigResponse`, `MarketplaceUnsubscribedMetricsRepository` (or equivalent DAO layer).
