# Expose Subscription Activation as `subscriptions[*].enabled`

## Status

Accepted

## Context

`GET /marketplace-overview/config` currently provides marketplace+region subscription rows but does not expose an explicit activation flag per row.  
Activation state exists in `org_market_mapping.enabled` (with `is_activated` as legacy fallback), and downstream clients need this state directly instead of inferring from row presence.

## Decision Drivers

- Provide explicit activation state per marketplace+region in API response.
- Keep compatibility with existing `org_market_mapping` source of truth.
- Support environments where legacy `is_activated` is still used.

## Considered Options

- Option 1: Keep current response without activation flag and infer in frontend.
- Option 2: Add `subscriptions[*].enabled` from `org_market_mapping.enabled` with fallback to `is_activated`.
- Option 3: Create a new endpoint only for activation state.

## Decision Outcome

Chosen option: "Option 2", because it adds minimal contract surface while preserving existing source and reducing client-side inference complexity.

### Positive Consequences

- Subscription activation is explicit and deterministic per row.
- Frontend can render enabled/disabled behavior without extra queries.
- Legacy compatibility is preserved via `is_activated` fallback mapping.

### Negative Consequences

- API contract changes for `ActiveSubscription` payload shape.
- Query semantics now include both enabled and disabled mappings (with flag), which may require frontend filtering logic.

## Pros and Cons of the Options

### Option 1

- Good: No API contract change.
- Bad: Clients must infer activation state and can mis-handle edge cases.

### Option 2

- Good: Clear, low-risk contract extension using existing data.
- Bad: Requires response model and docs updates.

### Option 3

- Good: Separation of concerns.
- Bad: Extra endpoint and additional round-trip for the same screen.

## More Information

- Impacted architecture sections: 6.1, 7.1, 7.5, 8.1, 15, 18.
- Impacted API: `GET /marketplace-overview/config`.
- Impacted backend modules: `MarketplaceOverviewRepository`, `MarketplaceOverviewConfigService`, `MarketplaceOverviewConfigResponse`.
