# ADR 24-03-2026: Brand Name Replacement in KPI Snapshot

## Status
Proposed (2026-03-24)

## Context
The prior architecture used `brand_id` as the primary brand identifier in the `marketplace_kpi_weekly_snapshot` table. However, the brand name ("brand") is already available in the BigQuery source tables (`viz_lost_sales`, etc.) and is often more stable for cross-system mapping and reporting. A requirement was raised to store the brand name ("brand") instead of the `brand_id` in the PostgreSQL snapshot table.

## Decision
1.  **Schema Update**: Replace the `brand_id` (BIGINT) column with a `brand` (VARCHAR) column in the `marketplace_kpi_weekly_snapshot` PostgreSQL table.
2.  **Aggregator Logic**: Update the `MarketplaceOverviewWeeklyAggregationTask` in `i2o-scheduler` to extract the `brand` column from BigQuery source tables and upsert it into the PostgreSQL sink.
3.  **API Filters**: Update the filter contract for `POST /widget/getCardData` and `POST /report/getGridCardData` to use `brand` (string values) instead of `brand_id`.
4.  **Uniqueness Constraint**: Update the unique constraint on `marketplace_kpi_weekly_snapshot` to include `brand` instead of `brand_id`.

## Technical Details
- **Column Change**: `brand_id BIGINT` → `brand VARCHAR(255)`
- **Constraint Change**: `UQ (org_id, week_start, marketplace, brand, region)`
- **Index Change**: `IDX (org_id, brand, region, week_start DESC)`

## Consequences
- **Positive**: Simplified mapping for reporting where brand names are the primary dimension.
- **Positive**: Reduced join overhead in some analytical queries where only the brand name is required.
- **Negative**: String-based lookups may have a slight performance impact compared to numeric IDs, but indexed VARCHAR filters in PostgreSQL are generally fast.
- **Negative**: Any existing data in the table will need a migration or flush/reload as part of the release.
