# ADR 20-03-2026-02: Precise Frontend Component and Auth Guard Alignment
 
## Status
Proposed (2026-03-20)
 
## Context
As part of the frontend architecture alignment for `marketplace-overview`, specific structural choices in the `architecture.md` document still referenced generic dependencies (e.g., bare PrimeNG Dropdowns) and incorrect Auth Guard names instead of the explicit internal components mandated by `SUMMARY-frontendapplication-i2oretail.md`.
 
To ensure the architecture serves as a precise blueprint for implementation, we must formally align the architecture with the project's internal `CoreModule` patterns.
 
## Decision
1. **Auth Guards**: The route `canActivate` array for the `MarketplaceOverviewModule` will use the explicitly named `KeycloakAuthenticationGuard` and `RolesBasedAuthGuard` implementations, conforming to the exact filenames in `src/app/auth/`.
2. **UI Component Encapsulation**: Instead of consuming PrimeNG widgets directly (e.g., `p-dropdown` for Brand/Region filters), the `FilterBarComponent` must explicitly use the shared filter components documented in the frontend summary (e.g., `app-multi-select-autocomplete` or `schedule-report-modal`) exposed via `CoreModule`.
3. **Module Wiring Documentation**: Added `CoreModule` as an explicit UI component dependency in the Integration Design (Section 10.1) to formalize this coupling to `src/app/modules/common/`.
 
## Consequences
- **Positive**: Complete alignment with the `frontendapplication-i2oretail` coding guidelines. Prevents engineers from implementing new ad-hoc PrimeNG widgets when common components already exist.
- **Positive**: Reduces boilerplate in the feature module by leveraging `CoreModule`.
- **Negative**: Adds a direct internal dependency to `CoreModule`, which is standard but couples the feature closely to the monolith's common UI definitions.
