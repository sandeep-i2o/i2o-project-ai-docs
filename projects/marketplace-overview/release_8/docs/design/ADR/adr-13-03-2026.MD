# Move Marketplace Snapshot And Scheduler Audit Tables To PostgreSQL

## Status

Accepted

## Context

The existing architecture stored `marketplace_kpi_weekly_snapshot` and `marketplace_scheduler_audit_log` in BigQuery.  
Release_8 update instruction requires both tables to be PostgreSQL tables with `org_id` for tenant-scoped persistence and audit traceability.

## Decision Drivers

- Release_8 directive: both snapshot and audit tables must be in PostgreSQL.
- Tenant isolation must remain mandatory through `org_id`.
- Scheduler observability and retry diagnostics require durable audit rows with relational indexing.

## Considered Options

- Option 1: Keep both tables in BigQuery (current design).
- Option 2: Move only audit log to PostgreSQL, keep snapshot in BigQuery.
- Option 3: Move both snapshot and audit tables to PostgreSQL.

## Decision Outcome

Chosen option: "Option 3", because it satisfies the explicit release requirement while preserving `org_id`-based data isolation and simplifying operational observability in one relational persistence layer.

### Positive Consequences

- Meets release requirement exactly.
- Unified PostgreSQL read/write path for dashboard snapshot + scheduler audit.
- Better point-lookups and incident diagnostics through relational indexes.

### Negative Consequences

- Requires DDL migration and rollout coordination across environments.
- Scheduler now depends on both BigQuery (source read) and PostgreSQL (sink write), increasing integration-test scope.

## Pros and Cons of the Options

### Option 1

- Good: No migration effort.
- Bad: Violates release requirement.

### Option 2

- Good: Partial operational improvement with lower migration impact.
- Bad: Does not fully satisfy requirement that snapshot must be in PostgreSQL.

### Option 3

- Good: Full requirement compliance and consistent persistence model.
- Bad: Highest implementation and migration effort.

## More Information

- Impacted architecture sections: 3, 4.1, 4.3, 5.3, 6.1, 6.3, 7.2, 10.1, 10.2, 11.2, 12, 13, 14.2, 15, 16, 17, 18.
- Affected modules: `i2o-scheduler`, `i2o-reseller` (data access + config), `i2o-master-data` (no schema change; still source for activation mapping).
- API impact: no contract change to `POST /widget/getCardData` and `POST /report/getGridCardData`; storage backing changed to PostgreSQL snapshot table.
- Data contract impact: mandatory `org_id` in `marketplace_kpi_weekly_snapshot` and `marketplace_scheduler_audit_log` with PostgreSQL indexes/uniqueness constraints.
- Migration note: apply PostgreSQL DDL before enabling scheduler writes in each environment.
