# Story: Database Schema & Flyway Migrations (fx_rates, sales_data, mv_daily_aggregates)

## Implementation Scope
Inside the `i2o-admin-db` PostgreSQL environment, execute structural database migrations using Flyway. This story initializes the `fx_rates` time-series (hyper) table utilizing the TimescaleDB extension, the static `sales_data` partition, and configures the heavy materialized view `mv_daily_aggregates` used for dashboard response compliance.

## Pre-requisites
- Local Docker environment mimicking GCP Cloud SQL Postgres + TimescaleDB.

## Acceptance Criteria
- [ ] Flyway script strictly written to create `fx_rates` table ensuring schema `date`, `currency_code`, `rate_to_eur`.
- [ ] TimescaleDB hypertable successfully partitioned by month for `fx_rates`.
- [ ] `sales_data` includes unique constraints `(date, region, sku)` to manage `ON CONFLICT UPSERT` behavior.
- [ ] The view `mv_daily_aggregates` handles grouping `sales_data` by `region`, `currency`, and `date`.
- [ ] Indexes exist: `date DESC, currency_code` on FX and `granularity, date, region` on the view.

## Execution Directives
- Drop SQL updates into: `sales-insights-api/src/main/resources/db/migration/` and `i2o-scheduler/` where applicable.
